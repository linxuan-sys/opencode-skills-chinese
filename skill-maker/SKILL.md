# 技能生成器

创建智能体技能并通过评估驱动的子代理循环迭代优化，
直到效果达到平台期或完成20次迭代。

## 概述

本技能将引导你完成智能体技能创建的全生命周期：

1. **捕获意图** - 理解技能需要实现的功能
2. **起草** - 编写SKILL.md和配套文件
3. **评估循环** - 生成子代理测试技能、对输出打分、检测
   平台期
4. **优化** - 根据评估反馈改进技能
5. **优化描述** - 调整技能描述以提升触发准确率

评估循环是核心流程：为每个测试用例生成独立的子代理，通过配套脚本对断言进行打分，汇总基准测试结果，不断迭代直到通过率达到平台期或完成20次迭代。

## 可用脚本

所有脚本使用Bun运行，执行命令：`bun run <path>`。

- **`scripts/grade.ts`** - 对照评估输出对断言进行打分
- **`scripts/aggregate-benchmark.ts`** - 将打分结果汇总到
  benchmark.json
- **`scripts/detect-plateau.ts`** - 检测迭代过程中的通过率平台期
- **`scripts/validate-skill.ts`** - 根据智能体技能规范验证SKILL.md
  文件的有效性
- **`scripts/optimize-description.ts`** - 优化技能描述以提升触发
  准确率
- **`scripts/eval-trigger.ts`** - 测试查询是否会触发技能
  描述
- **`scripts/update-history.ts`** - 跟踪迭代过程中的版本演进
- **`scripts/package-skill.ts`** - 打包技能用于分发
- **`eval-viewer/generate-review.ts`** - 生成静态HTML评估查看器

运行任意脚本时添加`--help`参数查看使用详情。

## 阶段1：捕获意图

在编写任何内容之前，先明确用户希望技能实现的功能。

### 需要明确的问题

1. 该技能应该让智能体能够完成什么任务？
2. 什么时候应该触发该技能？（用户话术、上下文、关键词）
3. 预期的输出格式是什么？
4. 是否有环境要求？（工具、软件包、网络）
5. 是否需要设置测试用例？（对于文件转换、数据提取、代码生成等可客观验证的输出需要设置，对于写作风格等主观输出可以跳过）

### 调研

在起草之前，先对领域进行调研：检查现有工具、识别边界用例、明确没有该技能时智能体不具备的上下文信息。在完全理解技能的用途、触发条件和成功标准之前，不要进入阶段2。

## 阶段2：起草技能

### 步骤1：创建技能目录

```bash
mkdir -p <skill-name>/scripts <skill-name>/references <skill-name>/assets <skill-name>/evals
```

### 步骤2：复制并填充模板

读取[assets/skill-template.md](assets/skill-template.md)并复制到`<skill-name>/SKILL.md`，填充所有`{{PLACEHOLDER}}`占位符。

参考[references/spec-summary.md](references/spec-summary.md)了解前言元数据约束和正文编写指南。

### 步骤3：编写描述

描述是技能的主要触发机制，决定了智能体是否会加载该技能。描述写得不好会导致技能永远不会被激活。

**规则：**

- 使用第三人称编写
- **必须同时包含**技能的功能 **和** "在...时使用"的触发条件
- 包含具体的触发关键词和同义词
- 可以稍微"主动"一点 - 智能体通常会触发不足，所以适当偏向更宽泛的触发条件
- 长度不超过1024字符
- **必须是单行文本** - 不要使用YAML多行标量（`>`或`|`），因为验证器中的极简YAML解析器会拒绝这种格式

**正面示例：**
`从PDF中提取文本、填写表单、合并PDF。在处理PDF文档或用户提到PDF、表单、文档提取时使用。`

**反面示例：** `分析git变更并生成符合约定的提交信息。`
(缺少"在...时使用"的描述——智能体不知道什么时候应该激活该技能)

### 步骤4：编写正文

遵循以下原则：

- **简洁是关键。** OpenCode很智能，只需要添加它不知道的上下文信息。
- **设置适当的自由度。** 对于脆弱的操作使用严格的指令，对于需要判断的任务使用灵活的指导。
- **解释原因。** 基于推理的指令（"执行X因为Y"）比僵化的指令（"始终执行X"）效果更好。
- **一个优秀的示例**胜过多个平庸的示例。
- **保持在500行以内。** 如果内容太长，拆分到参考文件中。
- **使用渐进式披露。** SKILL.md是概述内容；将大量参考资料（API文档、大型示例、查找表）移到`references/`目录下的文件中并添加链接。智能体会按需加载这些内容，保持基础上下文简洁。
- **包含工作流或检查清单。** 带有编号步骤或检查清单的技能比散文段落能产生更一致的结果。
- **添加"常见错误"部分。** 记录你见过或预期会出现的失败模式。当错误被明确列出时，智能体更擅长避免这些错误。

### 步骤5：按需添加脚本

如果技能涉及确定性操作（验证、数据处理、文件转换），在`scripts/`目录中打包脚本。脚本应该：

- 是自包含的，或者内联声明依赖
- 使用Bun作为首选运行时（添加`#!/usr/bin/env bun` shebang）
- 包含`--help`输出
- 使用结构化输出（标准输出返回JSON，标准错误返回诊断信息）
- 尽可能是幂等的
- 避免交互式提示

**始终使用Bun TypeScript (.ts)编写脚本**，除非技能的领域特别要求Python或其他运行时。Bun原生支持TypeScript、启动速度快、能自动安装依赖。不要默认使用Bash脚本——TypeScript脚本可维护性更高、错误处理更好，并且能自然生成结构化JSON输出。

对于有依赖的Bun脚本，在导入时固定版本（例如`import * as cheerio from "cheerio@1.0.0"`）。对于Python脚本（当领域要求时），使用PEP 723内联元数据并通过`uv run`运行。

### 步骤6：创建初始评估测试用例

在`<skill-name>/evals/evals.json`中编写2-3个符合实际场景的测试提示词。这些是验证技能是否有效的关键。即使在初始起草阶段，也要包含基本测试用例——后续可以再优化。格式详情请参见阶段3，但**不要跳过此步骤**。

```json
{
  "skill_name": "<skill-name>",
  "evals": [
    {
      "id": 1,
      "prompt": "符合实际场景的用户消息",
      "expected_output": "成功的标准是什么",
      "files": [],
      "assertions": []
    }
  ]
}
```

### 步骤7：验证（硬性门槛）

**在验证通过且零错误之前，不要进入阶段3。**

```bash
bun run scripts/validate-skill.ts <skill-dir>
```

修复所有错误，查看警告信息。常见验证失败原因：

- YAML多行描述（`>`或`|`）——改用单行值
- 名称包含大写字母——仅使用小写字母
- 名称与目录名不匹配——重命名目录或更新前言元数据
- 缺少描述——添加包含"在...时使用"触发条件的描述

---

## 阶段3：创建测试用例

在`evals/evals.json`中编写2-3个符合实际场景的测试提示词（格式参见阶段2步骤6，完整模式定义参见[references/schemas.md](references/schemas.md)）。

### 测试提示词质量检查清单

- 表述多样化（随意、精确、不同详细程度）
- 至少包含一个边界用例（格式错误的输入、不寻常的请求、模糊的指令）
- 符合实际上下文（文件路径、列名、个人上下文）
- 内容足够复杂，智能体确实能从技能中获益（不是简单的单步任务）

暂时不要编写断言——在阶段4运行评估时再起草。

---

## 阶段4：评估循环

这是技能创建流程的核心。你最多可以迭代20次，或者直到通过率达到平台期。

### 准备工作

在技能目录的同级目录下创建工作空间目录：

```bash
mkdir -p <skill-name>-workspace/iteration-1
```

### 每次迭代流程

#### 步骤1：生成子代理运行任务

对于evals.json中的每个评估，在同一轮中生成两个独立的子代理运行任务：

**使用技能的运行：**

```
执行以下任务：
- 阅读并遵循以下路径的技能：<path-to-skill>/SKILL.md
- 任务：<来自evals.json的评估提示词>
- 输入文件：<评估文件（如果有），否则填"none">
- 将所有输出保存到：<workspace>/iteration-<N>/eval-<name>/with_skill/outputs/
```

**基准运行**（相同提示词，不使用技能）：

```
执行以下任务（不使用技能）：
- 任务：<来自evals.json的评估提示词>
- 输入文件：<评估文件（如果有），否则填"none">
- 将所有输出保存到：<workspace>/iteration-<N>/eval-<name>/without_skill/outputs/
```

每个子代理必须从干净的上下文开始——不要保留之前运行的状态。这对于测试仅通过SKILL.md是否能提供足够的指导至关重要。

为每个评估目录编写`eval_metadata.json`文件：

```json
{
  "eval_id": 1,
  "eval_name": "描述性名称",
  "prompt": "评估提示词",
  "assertions": []
}
```

当**优化**现有技能时（迭代次数≥2），先对之前的版本打快照：

```bash
cp -r <skill-path> <workspace>/skill-snapshot/
```

然后让基准运行指向快照，使用`old_skill/`目录代替`without_skill/`。

#### 步骤2：在运行任务执行期间起草断言

在子代理运行任务执行的同时，为每个评估起草断言。好的断言应该：

- 可客观验证（"输出文件是有效的JSON"）
- 具体且可观察（"图表有带标签的坐标轴"）
- 可计数（"报告至少包含3条建议"）
- 测试**技能**带来的价值，而不是**提示词**提供的信息（如果提示词提到"600 DPI"，检查是否有"600"是在测试智能体的阅读理解能力，而不是技能的价值）

不好的断言：

- 太模糊无法打分（"输出很好"）
- 太脆弱（"输出完全使用短语'总营收：$X'"）
- 直接来自提示词（无论是否使用技能，智能体都会回显的关键词——这些断言在两种配置下都会通过）

将断言更新到`eval_metadata.json`和`evals/evals.json`中。

#### 步骤3：收集耗时数据

每个子代理完成后，立即将耗时数据保存到运行目录下的`timing.json`文件中：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这些数据来自任务完成通知，不会在其他地方持久化。每次运行完成后立即收集。

#### 步骤4：对输出打分

对每个完成的运行执行打分脚本：

```bash
bun run scripts/grade.ts <workspace>/iteration-<N>/eval-<name>/with_skill/
bun run scripts/grade.ts <workspace>/iteration-<N>/eval-<name>/without_skill/
```

该脚本会读取输出和断言，生成`grading.json`文件，包含每个断言的通过/失败结果和证据。

对于可以通过程序检查的断言（有效JSON、正确行数、文件存在），脚本会自动处理。对于需要判断的主观断言，使用[references/grader-prompt.md](references/grader-prompt.md)中的提示词生成评分子代理。评分器还会从输出中提取隐式声明、评估断言质量、提出评估改进建议——生成更丰富的grading.json文件。两种输出格式请参见[references/schemas.md](references/schemas.md)。

#### 步骤5：汇总基准测试结果

```bash
bun run scripts/aggregate-benchmark.ts <workspace>/iteration-<N> --skill-name <name>
```

该命令会生成`benchmark.json`和`benchmark.md`文件，包含每种配置的通过率、耗时和token使用情况，包括平均值、标准差和差值。

#### 步骤6：检测平台期

```bash
bun run scripts/detect-plateau.ts <workspace> --threshold 0.02 --window 2 --max-iterations 20
```

退出码含义：

- `0`（继续）：继续迭代
- `10`（平台期）：连续2次迭代通过率提升<2%，或通过率已达到100%。**停止迭代。**
- `20`（达到最大迭代次数）：已完成20次迭代。**停止迭代。**

如果状态是平台期或达到最大迭代次数，直接进入阶段5。

#### 步骤7：分析模式

在向用户展示结果之前，先分析基准测试数据：

- **无区分度的断言**：在两种配置下都总是通过。删除或替换这些断言。
- **总是失败的断言**：要么断言本身有问题，要么测试用例太难。修复这些断言。
- **高价值断言**：使用技能时通过，不使用时失败。理解背后的原因。
- **高方差评估**：不同运行的通过/失败结果不一致。收紧指令或修复不稳定的断言。
- **token/时间异常值**：如果某个评估的成本是其他评估的3倍，查看其执行记录找到瓶颈。

#### 步骤8：人工审核

向用户展示结果：

- 展示每个评估的通过率（使用技能 vs 基准）
- 展示总体提升幅度（技能带来的改进程度）
- 展示步骤7中得到的分析观察结果
- 询问用户对每个评估输出的反馈

记录反馈。没有反馈表示输出符合要求。

#### 步骤9：优化技能

你现在有三个信号来源：

1. **失败的断言** - 技能的具体不足
2. **人工反馈** - 更广泛的质量问题
3. **执行记录** - 问题出现的原因

使用这三个来源优化技能。核心原则：

- **从反馈中提炼通用规则。** 技能会被用于很多不同的提示词，而不仅仅是这些测试用例。避免过拟合到特定示例。
- **保持技能精简。** 更少但更精准的指令通常比详尽的规则效果更好。如果执行记录显示存在无用工作，删除相关指令。
- **解释原因。** "执行X因为Y容易导致Z"比"始终执行X，永远不要执行Y"效果更好。
- **封装重复工作。** 如果每次测试运行都独立编写了类似的辅助脚本，将其打包到`scripts/`目录中。

对技能应用优化措施，创建新的迭代目录，回到步骤1继续迭代。

#### 高级功能：盲测对比（可选）

对于严格的版本对比，使用盲测A/B对比消除偏差：使用[references/comparator-prompt.md](references/comparator-prompt.md)中的提示词生成对比子代理，传入未标记的输出，然后使用分析器（[references/analyzer-prompt.md](references/analyzer-prompt.md)）解释胜出的原因。当迭代之间的通过率接近，或者你需要结构化的改进原因分析时使用此功能。

## 阶段5：最终交付

### 验证最终技能

```bash
bun run scripts/validate-skill.ts <skill-dir>
```

### 优化描述

在技能内容稳定后，优化技能描述以提升触发准确率。

1. 生成20个评估查询 - 混合应该触发（8-10个）和不应该触发（8-10个）的查询：
   - 应该触发：技能处理任务的不同表述，包括间接引用
   - 不应该触发：共享关键词但需要不同工具的近似查询。不要使用明显不相关的查询。

2. 对每个查询，测试技能的描述是否会让智能体选择该技能
3. 调整描述以提升真阳性率，降低假阳性率
4. 重新测试直到满意

### 安装技能

```bash
cp -r <skill-name> ~/.agents/skills/<skill-name>   # 跨客户端安装
cp -r <skill-name> .agents/skills/<skill-name>      # 项目级安装
```

### 最终检查清单

- [ ] SKILL.md有有效的前言元数据（名称、描述）
- [ ] 名称与目录名一致
- [ ] 描述包含功能 + 触发时机 + 触发关键词
- [ ] 正文在500行以内
- [ ] 脚本有--help说明、结构化输出、有意义的退出码
- [ ] 至少有3个带断言的评估测试用例
- [ ] 评估循环运行结果显示相比基准有可衡量的提升
- [ ] 没有缺失引用的文件
- [ ] 所有脚本都能通过`bun run`成功运行

## 环境说明
